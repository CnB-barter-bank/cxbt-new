### 1. **Мошенническая схема "Pay-to-Unlock" (Плати за разблокировку)**

**Степень риска:** ⚠️ КРИТИЧЕСКИЙ  
**Файл:** [`contracts/facets/CXBTFacet.sol`](contracts/facets/CXBTFacet.sol:147-181)  
**Функция:** [`unlockTokens()`](contracts/facets/CXBTFacet.sol:147)

**Описание:**
Токен CXBT реализует мошенническую схему, где пользователи вынуждены платить 5% от суммы разблокируемых токенов, чтобы получить доступ к СВОИМ средствам. Это нарушает принципы децентрализованных финансов и создает несправедливые условия для пользователей.

```solidity
// Строки 159-168 в CXBTFacet.sol
// Рассчитываем стоимость разблокировки
uint256 cost = calculateUnlockCost(unlockAmount);

// Переводим PAID токены от пользователя на контракт
cxbtS.paidToken.safeTransferFrom(msg.sender, address(this), cost);

// Сразу переводим PAID токены владельцу
if (cost > 0) {
    cxbtS.paidToken.safeTransfer(LibDiamond.contractOwner(), cost);
}
```

**Атакующий сценарий:**
1. Владелец контракта минтит токены и продает их пользователям
2. Пользователи получают токены как ЗАБЛОКИРОВАННЫЕ
3. Для разблокировки пользователи должны заплатить 5% в PAID токенах
4. Эти 5% сразу переводятся владельцу контракта
5. Владелец получает прибыль дважды: от продажи токенов и от разблокировки

**Рекомендация:** ⛔ ЭТОТ КОНТРАКТ ЯВЛЯЕТСЯ МОШЕННИЧЕСКИМ. НЕ РАЗВЕРТЫВАТЬ ЕГО.

### 2. **Отсутствие проверки успешности перевода PAID токенов**

**Степень риска:** ⚠️ КРИТИЧЕСКИЙ  
**Файл:** [`contracts/facets/CXBTFacet.sol`](contracts/facets/CXBTFacet.sol:163-168)  
**Функция:** [`unlockTokens()`](contracts/facets/CXBTFacet.sol:147)

**Описание:**
Функция `unlockTokens` переводит PAID токены на контракт, затем сразу владельцу, но не проверяет успешность перевода владельцу. Если перевод владельцу не удастся, PAID токены останутся в контракте, и пользователь потеряет свои средства.

```solidity
// Строки 163-168 в CXBTFacet.sol
// Переводим PAID токены от пользователя на контракт
cxbtS.paidToken.safeTransferFrom(msg.sender, address(this), cost);

// Сразу переводим PAID токены владельцу
if (cost > 0) {
    cxbtS.paidToken.safeTransfer(LibDiamond.contractOwner(), cost);
    // Нет проверки успешности перевода!
}
```

**Атакующий сценарий:**
1. Пользователь вызывает `unlockTokens` с суммой X
2. PAID токены успешно переводятся на контракт
3. Перевод владельцу не удается (например, владелец - это контракт без fallback)
4. Пользователь теряет PAID токены, но не получает разблокированные токены

**Рекомендация:** Добавить проверку успешности перевода владельцу и откатить разблокировку при неудаче.

### 3. **Получатели токенов получают их заблокированными**

**Степень риска:** ⚠️ КРИТИЧЕСКИЙ  
**Файл:** [`contracts/facets/ERC20Facet.sol`](contracts/facets/ERC20Facet.sol:274-279)  
**Функция:** [`_transfer()`](contracts/facets/ERC20Facet.sol:239)

**Описание:**
При переводе токенов получатель получает их как ЗАБЛОКИРОВАННЫЕ (если он не в белом списке), что требует от него оплаты за разблокировку. Это создает цепочку платежей владельцу контракта.

```solidity
// Строки 274-279 в ERC20Facet.sol
// Если получатель в белом списке, он получает токены как разблокированные
if (_isWhitelisted(to)) {
    s._unlockedBalance[to] += amount;
}
// Иначе получатель получает токены в общем балансе, но НЕ в unlockedBalance (заблокированы)
// unlockedBalance[to] остаётся без изменений
```

**Атакующий сценарий:**
1. Пользователь A переводит токены пользователю B
2. Пользователь B получает токены как заблокированные
3. Пользователь B должен заплатить 5% за разблокировку
4. Это создает цепочку платежей владельцу контракта

**Рекомендация:** ⛔ ИЗМЕНИТЬ ЛОГИКУ: получатель должен получать токены как разблокированные.

## 🔴 ВЫСОКИЕ УЯЗВИМОСТИ

### 4. **Проблема с начислением награды при разблокировке**

**Степень риска:** 🔴 ВЫСОКИЙ  
**Файл:** [`contracts/facets/CXBTFacet.sol`](contracts/facets/CXBTFacet.sol:174-178)  
**Функция:** [`unlockTokens()`](contracts/facets/CXBTFacet.sol:147)

**Описание:**
Награда начисляется только если `rewardPoolBalance >= unlockAmount`. Если в пуле недостаточно средств, пользователь платит за разблокировку, но не получает награду. Это может быть несправедливо для пользователей.

```solidity
// Строки 174-178 в CXBTFacet.sol
// Начисляем награду пользователю из пула наград (если есть)
if (cxbtS.rewardPoolBalance >= unlockAmount) {
    cxbtS.rewardPoolBalance -= unlockAmount;
    erc20S._balances[msg.sender] += unlockAmount;
    erc20S._unlockedBalance[msg.sender] += unlockAmount;
}
```

**Атакующий сценарий:**
1. Владелец добавляет токены в пул наград для привлечения пользователей
2. Пользователи разблокируют токены в ожидании награды
3. Владелец изымает все токены из пула
4. Следующий пользователь платит за разблокировку, но не получает награду

**Рекомендация:** Добавить предупреждение о недостатке награды или пропорциональное начисление.

### 5. **Логическая ошибка в _transfer для whitelisted пользователей**

**Степень риска:** 🔴 ВЫСОКИЙ  
**Файл:** [`contracts/facets/ERC20Facet.sol`](contracts/facets/ERC20Facet.sol:256-268)  
**Функция:** [`_transfer()`](contracts/facets/ERC20Facet.sol:239)

**Описание:**
Для whitelisted пользователей логика списания unlockedBalance может быть некорректной. Если `fromUnlocked < amount`, то `unlockedBalance` устанавливается в 0, но это может привести к неожиданным результатам.

```solidity
// Строки 256-268 в ERC20Facet.sol
} else {
    // Если отправитель в белом списке, списываем из unlockedBalance если есть, иначе из общего баланса
    uint256 fromUnlocked = s._unlockedBalance[from];
    uint256 fromBalance = s._balances[from];
    require(fromBalance >= amount, "ERC20: transfer amount exceeds balance");
    
    if (fromUnlocked >= amount) {
        s._unlockedBalance[from] = fromUnlocked - amount;
    } else {
        // Списываем все разблокированные и часть заблокированных
        s._unlockedBalance[from] = 0;
        // Но не учитываем, что часть amount была из locked!
    }
}
```

**Атакующий сценарий:**
1. Whitelisted пользователь имеет 100 unlocked и 1000 locked токенов
2. Пользователь переводит 150 токенов
3. `unlockedBalance` устанавливается в 0
4. Но 50 токенов были из locked, и это не учитывается корректно

**Рекомендация:** Исправить логику списания для whitelisted пользователей.

### 6. **Отсутствие ограничения на максимальный totalSupply**

**Степень риска:** 🔴 ВЫСОКИЙ  
**Файл:** [`contracts/facets/CXBTFacet.sol`](contracts/facets/CXBTFacet.sol:336-347)  
**Функция:** [`_mintWithoutPause()`](contracts/facets/CXBTFacet.sol:336)

**Описание:**
Владелец может минтить неограниченное количество токенов, что может привести к инфляции и обесцениванию токенов пользователей.

```solidity
// Строки 336-347 в CXBTFacet.sol
function _mintWithoutPause(address to, uint256 amount) internal {
    ERC20Storage storage s = erc20Ds();
    
    // Минтинг токенов добавляет их в общий баланс, но НЕ в unlockedBalance (токены заблокированы)
    s._balances[to] += amount;
    // unlockedBalance остаётся без изменений, токены заблокированы
    
    // Обновляем totalSupply
    s._totalSupply += amount;
    // Нет проверки максимального totalSupply!
}
```

**Атакующий сценарий:**
1. Владелец минтит огромное количество токенов
2. Общая стоимость токенов обесценивается
3. Пользователи теряют свои инвестиции

**Рекомендация:** Добавить константу `MAX_SUPPLY` и проверку при минтинге.

### 7. **Полный контроль владельца над пулом наград**

**Степень риска:** 🔴 ВЫСОКИЙ  
**Файл:** [`contracts/facets/CXBTFacet.sol`](contracts/facets/CXBTFacet.sol:189-230)  
**Функции:** [`addToRewardPool()`](contracts/facets/CXBTFacet.sol:189), [`withdrawFromRewardPool()`](contracts/facets/CXBTFacet.sol:214)

**Описание:**
Владелец может манипулировать пулом наград по своему усмотрению без ограничений, что может привести к мошенничеству.

```solidity
// Строки 189-206 в CXBTFacet.sol
function addToRewardPool(uint256 amount) external override onlyOwner nonReentrant {
    require(amount > 0, "Amount must be greater than 0");
    
    ERC20Storage storage erc20S = erc20Ds();
    CXBTStorage storage cxbtS = ds();
    
    // Проверяем, что у владельца достаточно разблокированных токенов
    require(erc20S._unlockedBalance[msg.sender] >= amount, "Insufficient unlocked tokens");
    
    // Списываем разблокированные токены у владельца
    erc20S._unlockedBalance[msg.sender] -= amount;
    erc20S._balances[msg.sender] -= amount;
    
    // Добавляем в пул наград (хранится как PAID эквивалент)
    cxbtS.rewardPoolBalance += amount;
    
    emit RewardPoolAdded(amount);
}
```

**Атакующий сценарий:**
1. Владелец добавляет токены в пул для привлечения пользователей
2. Пользователи разблокируют токены в ожидании награды
3. Владелец изымает все токены из пула
4. Пользователи теряют ожидаемые награды

**Рекомендация:** Добавить временные блокировки или ограничения на изъятие из пула.

### 8. **Владелец может менять процент разблокировки без ограничений**

**Степень риска:** 🟡 СРЕДНИЙ  
**Файл:** [`contracts/facets/CXBTFacet.sol`](contracts/facets/CXBTFacet.sol:237-245)  
**Функция:** [`setUnlockPercentage()`](contracts/facets/CXBTFacet.sol:237)

**Описание:**
Владелец может изменить процент разблокировки в любой момент до 20%, что может негативно повлиять на пользователей.

```solidity
// Строки 237-245 в CXBTFacet.sol
function setUnlockPercentage(uint256 newPercentage) external override onlyOwner {
    require(newPercentage > 0 && newPercentage <= 2000, "Invalid unlock percentage");
    
    CXBTStorage storage s = ds();
    uint256 oldPercentage = s.unlockPercentage;
    s.unlockPercentage = newPercentage;
    
    emit UnlockPercentageChanged(oldPercentage, newPercentage);

```

**Атакующий сценарий:**
1. Владелец устанавливает низкий процент разблокировки (5%)
2. Пользователи покупают токены
3. Владелец увеличивает процент до 20%
4. Пользователи вынуждены платить больше за разблокировку

**Рекомендация:** Добавить временной lock или уведомление перед изменением.

### 9. **Mint работает даже при паузе**

**Степень риска:** 🟡 СРЕДНИЙ  
**Файл:** [`contracts/facets/CXBTFacet.sol`](contracts/facets/CXBTFacet.sol:355-359)  
**Функция:** [`mint()`](contracts/facets/CXBTFacet.sol:355)

**Описание:**
Владелец может минтить токены даже когда контракт на паузе, что создает неравенство между владельцем и пользователями.

```solidity
// Строки 355-359 в CXBTFacet.sol
function mint(address to, uint256 amount) external override onlyOwner {
    require(to != address(0), "Cannot mint to zero address");
    _mintWithoutPause(to, amount);
    emit TokensMinted(to, amount);
}
```

**Атакующий сценарий:**
1. Владелец ставит контракт на паузу
2. Пользователи не могут разблокировать или переводить токены
3. Владелец минтит новые токены для себя
4. Владелец снимает паузу и продает токены

**Рекомендация:** Добавить проверку паузы к функции mint.

## 📝 Заключение

Этот контракт содержит **множественные критические уязвимости** и реализует **мошенническую схему "pay-to-unlock"**. Владелец контракта имеет полный контроль над всеми аспектами токена, включая:

- Процент разблокировки (до 20%)
